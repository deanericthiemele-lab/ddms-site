Fiche N°1 — Coupure de Lien (Node/Interface Down)
Détection · Vérification · Escalade SAV B2B
P1 / P2 ▼
DÉCLENCHEUR : Alerte SolarWinds "Node Down" ou "Interface Down" — Délai max de prise en charge : 5 min (P1)
Procédure de traitement
1

DÉTECTION

Identifier l'alerte sur SolarWinds NPM (Node Down / Interface Down). Noter l'heure, le client, la référence du lien.

2

VÉRIFICATION

Consulter NPM → confirmer l'état du nœud/interface et la durée de l'indisponibilité. Vérifier que ce n'est pas un faux positif (poll unique).

3

APPEL CLIENT

Contacter le client pour confirmer la coupure côté site. Demander : état équipements (routeur, ONT), voyants lumineux, coupures électriques éventuelles.

4

DIAGNOSTIC

Vérifier avec le client : état du routeur CE, de l'ONT, alimentation électrique. Tenter un ping depuis un autre équipement du réseau.

5

CONFIRMATION

Lien confirmé DOWN après appel + vérification → ouvrir ticket ITSM immédiatement. Notifier le client (Template C-01 si P1).

6

ESCALADE SAV B2B

Envoyer email d'escalade SAV B2B avec tous les éléments requis (voir template ci-dessous).

Fiche N°3 — VPS / Serveur Privé Virtuel (Cloud OCI)
CPU · RAM · Disque · Injoignable · Latence · Billing
P1 / P2 ▼
DÉCLENCHEUR : Alerte SolarWinds NPM ou portail OCI — Prise en charge <10 min (P1)
Procédure de traitement
1

DÉTECTION

Identifier l'alerte sur SolarWinds ou portail OCI. Relever : nom VPS, IP publique, client, type d'alerte.

2

VÉRIFICATION

Se connecter au portail OCI. Vérifier statut VPS (running/stopped/error). Contrôler CPU, RAM, disque en temps réel.

3

DIAGNOSTIC

Identifier la cause : pic CPU, saturation RAM, disque plein, panne réseau, billing anormal. Consulter les logs si accessible.

4

ACTION

Appliquer l'action corrective selon le tableau ci-dessous. Documenter chaque action dans le ticket ITSM.

5

VÉRIFICATION POST-ACTION

Surveiller les métriques 15 min minimum. Confirmer retour à la normale sur le portail OCI.

6

ESCALADE / OCCS

Si non résolu en 20 min ou VPS injoignable sans cause identifiable → ouvrir requête OCCS + escalader N2.

7

COMMUNICATION CLIENT

Notifier le client si impact service confirmé. Mise à jour toutes les 30 min. Clôturer proprement dans ITSM.

Fiche N°5 — Email Pro / Messagerie (Cloud OCI)
SMTP · DKIM · SPF · Stockage · Webmail · Certificats
P1 / P2 ▼
DÉCLENCHEUR : SMTP indisponible, DKIM en échec, taux rebond >5%, stockage >70%
Procédure de traitement
1

DÉTECTION

Identifier l'alerte (monitoring OCI ou signalement client). Relever : domaine, type de problème, heure.

2

VÉRIFICATION

Vérifier statut service Email sur portail OCI. Tester envoi/réception depuis un compte test.

3

DIAGNOSTIC SMTP

SMTP KO → vérifier statut serveur portail OCI. Tester port 25/465/587 depuis l'extérieur.

4

DIAGNOSTIC DNS/AUTH

DKIM/SPF/DMARC en échec → vérifier enregistrements DNS TXT du domaine via dig ou MXToolbox.

5

DIAGNOSTIC RÉPUTATION

Taux rebond >5% → vérifier blacklists IP (MXToolbox, Spamhaus). Analyser headers des mails rejetés.

6

ACTION CORRECTIVE

Appliquer action selon le tableau. Alerter client si intervention de sa part requise (nettoyage boîtes, renouvellement).

7

ESCALADE / OCCS

Si service email KO côté infrastructure OCI → requête OCCS immédiate + ticket Orange Cloud.
